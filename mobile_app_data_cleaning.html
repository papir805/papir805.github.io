Case Study - Part 1: Data Cleaning

Goal: To clean mobile app gaming data in preparation for analysis.

PostgreSQL
Data Cleaning

Background

This article covers part 1 and is meant to detail my process as I progress through the cleaning part of the case study. I'll discuss the obstacles that I encounter, how I overcome them, and my overall logic in general.

Information on the game and the case study can be found here. The full project containing the code and both the original and cleaned versions of the data can be found on my GitHub.

Section 1: Loading the Dataset

The dataset comes in two csv files, users.csv and event_performance.csv

users.csv

This file contains a complete list of every player that has an account for the game. It also contains information about user attributes.
Each line represents a unique user.

Field Description
userid A string that uniquely identifies a player
subscriber A binary variable identifying whether the userid is a paid subscriber or not
category A variable that identifies whether the userid falls in category 'A', 'B', or 'C'

event_performance.csv

This file contains the total points scored during every hour of gameplay over the course of 2019 for every participating player.
Each line represents the number of points a user scored during a single hour of gameplay.

Field Description
userid A string that uniquely identifies a player
event_date The date corresponding to when the userid participated in a gaming event
hour The hour of the day in which the userid participated in the gaming event
points The number of points scored by the userid during a single hour of gameplay

Because the data may need cleaning, I create two staging tables with each field as text.  Using the text data type is the safest as the data is coming from a CSV file.  This will hopefully help the data get ingested into the database without any errors, so I can begin inspecting it.  After creating the tables, the data is ingested into them from the CSV files. 
    DROP TABLE IF EXISTS users_staging;
    CREATE TABLE users_staging (userid text, subscriber text, category text);
    COPY users_staging(userid, subscriber, category)
    FROM '{working_dir}/data/users.csv'
    DELIMITER ','
    CSV HEADER;									
    DROP TABLE IF EXISTS event_performance_staging;
    CREATE TABLE event_performance_staging (userid text, event_date text, hour text, points text);
    COPY event_performance_staging(userid, event_date, hour, points)
    FROM '{working_dir}/data/event_performance.csv'
    DELIMITER ','
    CSV HEADER;
Now to start inspecting and cleaning. After the data is cleaned, I'll create new tables with more explicit data types and populate them with it.

Section 2: Checking for Null Values 

 SELECT userid, subscriber, category
    FROM users_staging
 WHERE userid IS NULL
         OR subscriber IS NULL
         OR category IS NULL;						
 SELECT userid, event_date, hour, points
    FROM event_performance_staging
  WHERE userid IS NULL
         OR event_date IS NULL
         OR hour IS NULL
         OR points IS NULL;
Both queries return nothing, indicating there are no null values in any of the columns in either of the tables.

Section 3: Checking for Duplicate Values 

          SELECT userid, COUNT(userid)
             FROM users_staging
    GROUP BY userid
        HAVING COUNT(userid) > 1;					
          SELECT userid, event_date, hour, points, COUNT(*)
             FROM event_performance_staging
    GROUP BY userid, event_date, hour, points
        HAVING COUNT(*) > 1;
Both queries return nothing, indicating there are no duplicate values in either of the tables.

Section 4: Cleaning the users_staging Table


userid

Often, userids are standardized in some way and have a structure, but no information on that was provided in the case study. 
         SELECT LENGTH(userid) AS userid_length
                      , COUNT(*) AS frequency
             FROM users_staging
    GROUP BY LENGTH(userid);

This query checks to see if userids have a standardized length and reveals all userids are 36 characters long.  A visual inspection of the CSV file shows userids contain a mixture of alphanumeric characters.  I'll make them VARCHAR(36) when moving the cleaned data to the new table.

subscriber

Because subscriber is a binary variable, I would like it to be the int data type. 
  SELECT DISTINCT(subscriber::int)
     FROM users_staging;

It converts to int with no problems, and all values in the dataset are either 0 or 1, as they're supposed to be.  A value of 0 means the user isn't a subscriber, while a value of 1 means they are a subscriber.  Everything looks good here.

category

Because category is a single character string representing which category a user falls into, I would like it to be the VARCHAR(1) data type. 
  SELECT DISTINCT(category::VARCHAR(1))
     FROM users_staging;

It converts to VARCHAR(1) with no problems, and all values in the dataset are either 'A', 'B', or 'C', as they're supposed to be.  Everything looks good here too.

Section 5: Cleaning the events_performance_staging Table


userid

It was noted earlier that all userids in the users_staging table had 36 characters. 
         SELECT LENGTH(userid) AS userid_length
                      , COUNT(*) AS frequency
             FROM event_performance_staging
    GROUP BY LENGTH(userid);

However, two userids in the events_performance_staging table have 37 characters.  This has to be a mistake, and I isolated those rows to figure out why. 

     SELECT userid
                 , LENGTH(userid) AS userid_length
       FROM event_performance_staging
    WHERE LENGTH(userid) = (SELECT MAX(LENGTH(userid)) 
                                                       FROM event_performance_staging);

It appears one userid had a whitespace character at the end (denoted by "\s") and the other had a double quotation mark. Later, I'll use REGEX to trim both of those characters when importing to the new tables.

event_date

The event_date field has the most obstacles to overcome.  It contains several different forms of dates that aren't valid, as well as inconsistent date formats. 

DELETE FROM event_performance_staging
            WHERE event_date = '19/24/2019';

One of the dates is 19/24/2019.  Since there's no way to identify whether this date is meant to be 1/24/2019, 9/24/2019, or perhaps something else entirely, the safest option is to drop it.  Dropping one entry in a dataset this size shouldn't make much of a difference. 


ALTER TABLE event_performance_staging
	ALTER COLUMN event_date
	        TYPE DATE
	                USING (to_date(event_date, 'MM/DD/YY'));

Because some event_dates use a 2 digit year format (MM/DD/YY) while other event_dates use a different 4 digit year format (YYYY/MM/DD), I'm also going to use the to_date function to convert any dates from the MM/DD/YY format to the more standard YYYY/MM/DD format.

Lastly, there are two dates remaining that are invalid and need to be dropped. 

WITH first_dates AS (
SELECT event_date
             , ROW_NUMBER() OVER (ORDER BY event_DATE ASC) AS rn
    FROM event_performance_staging
     LIMIT 3
),
last_dates AS (
SELECT event_date
             , ROW_NUMBER() OVER (ORDER BY event_DATE DESC) AS rn
    FROM event_performance_staging
     LIMIT 3
)
       SELECT fd.event_date
                    , ld.event_date
           FROM first_dates AS fd
 INNER JOIN last_dates as ld
                ON fd.rn = ld.rn
  ORDER BY fd.event_date ASC, ld.event_date ASC;

There is one date from the future, 2039-08-02, and one date from too far in the past to be valid, 1999-03-22. The rows corresponding to these dates will have to be dropped when I import the cleaned data to the new tables.


hour

Because hour represents an hour of the day, I would like to convert it to the int data type in the new table. 

        SELECT DISTINCT(hour::int)
            FROM event_performance_staging
    ORDER BY 1;

It converts just fine, and the values appear to be using military time, ranging from 16:00 to 20:00 or 4:00 to 8:00 p.m.


points

Attempting to cast points to int results in an error and indicates one of the point scores has a quotation mark in it. Other point scores could also have letters or special characters, so let's see if there are any others with issues. 
SELECT points
FROM event_performance_staging
WHERE points ~ '^.*[^A-Za-z0-9 .-].*$';

Using REGEX, I can identify which rows are problematic. In addition to the point score that had quotation marks, another point score has question marks. Later, I'll use REGEX to trim those characters and then cast the points column to int in the new table.

Section 6: Moving Cleaned Data to New Tables


users

No cleaning was really needed for the users_staging data. The query below simply creates a new table called users with more explicit data types and introduces constraints to ensure all rows get entered with the correct values. 

DROP TABLE IF EXISTS users;

CREATE TABLE users (
	  userid VARCHAR(36) NOT NULL,
  subscriber int NOT NULL,
  category VARCHAR(1) NOT NULL,
 PRIMARY KEY (userid),
  CONSTRAINT valid_subscriber CHECK (subscriber IN (0, 1)),
  CONSTRAINT valid_category CHECK (category IN ('A', 'B', 'C'))
	);

 INSERT INTO users(userid, subscriber, category)
      SELECT userid
           , subscriber
           , category
        FROM users_staging;
I plan to set up the following schema for a newly defined users table:
  1. userid
    • Data type: VARCHAR(36)
    • Constraints: PRIMARY KEY
  2. subscriber
    • Data type: int
    • Constraints: only non-null values of 0 or 1 are allowed
  3. category
    • Data type: VARCHAR(1)
    • Constraints: only non-null values of 'A', 'B', or 'C' are allowed

Once the new table is created, the query then copies the data from users_staging table to the new users table.

event_performance

The event_performance_staging table needed a little bit of cleaning. The query below creates a new table called event_performance with more explicit data types and introduces constraints, but also uses REGEX to remove typos from the userid and points fields, as well as using a WHERE clause to filter out observations corresponding to invalid dates. 

DROP TABLE IF EXISTS event_performance;

CREATE TABLE event_performance (
	    userid VARCHAR(36) NOT NULL,
	event_date date NOT NULL,
	      hour int NOT NULL,
	    points int NOT NULL,
	CONSTRAINT valid_hour CHECK (hour >= 0 AND hour <= 23),
    CONSTRAINT valid_event_date CHECK (event_date <= CURRENT_DATE AND 
                                       event_date >= '2013-01-01')
	);

INSERT INTO event_performance(userid, event_date, hour, points) 
     SELECT REGEXP_REPLACE(userid, '[" ]', '', 'gi')
          , event_date
          , hour
          , REGEXP_REPLACE(points, '["?]', '', 'gi')::int
       FROM event_performance_staging
      WHERE event_date <= '2023-07-13' 
        AND event_date >= '2013-01-01';
I plan to set up the following schema for a newly defined events_performance table:
  1. userid
    • Data type: VARCHAR(36)
    • Constraints: can't be null
    • Note: I use REGEX to remove the whitespace character and the quotation marks from the userids that were 37 characters long
  2. event_date
    • Data type: date
    • Constraints: Must be a non-null date from after the company was formed (2013-01-01) and before the current date.
    • Note: I filter the dates that were identified to be invalid from earlier.
  3. hour
    • Data type: int
    • Constraints: must be a non-null integer from 0 to 23 inclusive
  4. points
    • Data type: int
    • Constraints: can't be null

Once the new table is created, the query then copies the data from the event_performance_staging table to the new event_performance table.
