Part 3: Correlation Between User Attributes and Yearly Point Totals

Goal: To explore and quantify correlation between user attributes and yearly point totals through regression and hypothesis testing.

Python, PostgreSQL
Visualization, Regression, and Hypothesis Testing

Section 1: Background

In part 2 of this project, I queried mobile gaming data from a PostgreSQL database and performed an exploratory data analysis trying to understand user performance. This part of the project explores if there's a correlation between user attributes and user performance, however, some background on the data itself may be helpful before proceeding.

The data comes from a mobile game produced by a gaming company that runs periodic gaming events. Gaming events last several hours and happen at most once a day. During a gaming event, all players play cooperatively as the human team to earn points against an enemy team comprised of a network of AI bots. Individual players on each team win or lose points when certain actions are taken in game; however, the total point score of each team at the end of the gaming event is all that matters for the win condition. If the humans have more points when the event ends, then the humans win; otherwise, the enemy AI team wins.

To entice users to continue playing, there are rewards for winning a gaming event, but to keep things interesting, there are also penalties for losing. When the human team wins, users win prizes based on a tiered system where higher tiers earn better prizes and the point total at game's end determines which tier's prizes are won. On the other hand, should the game end and the human team has a negative score, then all players are forced to take a penalty. Like prizes, penalties have a tiered structure too, which gets progressively more and more severe for worse and worse tiers.

User Table

This table contains a list of all players for the game and some information about user attributes. Each row in the table represents a unique user.

Field SQL Data Type Description
userid VARCHAR(36) A 36 character string that uniquely identifies a player
attribute1 INT A binary variable identifying whether a user has attribute1 or not
attribute2 VARCHAR(1) A variable that identifies whether a user falls in category 'A', 'B', or 'C'

Event Performance Table

This table contains an hour-by-hour breakdown of the points earned for each player during every gaming event of 2019. Each row in the table represents the number of points scored by a userid during a single hour of gameplay of a gaming event.

Field SQL Data Type Description
userid VARCHAR(36) A 36 character string that uniquely identifies a player
event_date DATE The date the user participated in the gaming event
hour INT The hour of the day in which the user participated in the gaming event
points INT The number of points scored by a particular user during a single hour of a gaming event

The full project with the original and cleaned versions of the data can be found on my GitHub. Now to start checking for correlation...

Section 2: Visualizing Performance, Segmented by User Attributes

SQLAlchemy, Pandas, Matplotlib, and Seaborn

For the purposes of this project, I'll define a user's performance to be their yearly point total and explore if there's any correlation with the value of attribute1 or attribute2.

Joining Tables

WITH total_points_per_user AS (
   SELECT userid
        , SUM(points) AS total_points
     FROM event_performance
 GROUP BY userid
  )

   SELECT u.userid
        , u.attribute1
        , u.attribute2
        , COALESCE(tp.total_points, 0) AS total_points
     FROM users AS u
LEFT JOIN total_points_per_user AS tp
       ON u.userid = tp.userid;

Because information on attribute1 and attribute2 are in the users table, but information on point scores are in the event_performance table, the tables have to be joined before any investigation on correlation can happen. Because there may be users in the users table who have never participated in a gaming event and thus have no entries in the event_performance table, I perform a LEFT JOIN with the users table. This will ensure all userids from the users table end up in the joined table. COALESCE will force any NULL values resulting from userids users table with no corresponding entries in the event_performance table to 0.
